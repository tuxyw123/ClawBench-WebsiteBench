# **Web2Code2Web 执行计划 v0.1**

## **一、项目定义**

**Web2Code2Web \= Web → Code → Web。**

给 Agent 一个由我们自己部署的网站。Agent 不能获得目标网站源代码，只能通过 `browser-use` 一类的浏览器工具进行探索，随后输出一套可编译、可运行的前后端代码。系统重新部署这套代码，并从用户体验、交互行为和后端业务逻辑三个层面，与目标网站进行对比。

核心不是“代码长得一样”，而是：

在等价初始状态下，用户执行同一组操作时，目标网站和复刻网站产生尽可能一致的页面、反馈、状态变化和最终业务结果。

例如，一个电商站的完整体验应包括：

* 注册、登录、退出；  
* 忘记密码和重置密码；  
* 搜索、筛选、商品详情；  
* 加购物车、修改数量、结算；  
* 下单、查看订单；  
* 刷新或重新登录后状态仍然存在；  
* “同一用户五分钟内只能注册一次”等业务约束。

## **二、先冻结的关键决策**

| 问题 | v1 决策 |
| ----- | ----- |
| Agent 能否拿到目标源代码 | **不能**。源代码只供 Corpus 构建者和评测系统使用 |
| Agent 的输入 | 网站 URL、公开 PRD、测试账号、工具和资源预算 |
| Agent 如何探索 | 通过受控的 browser-use 接口操作网站 |
| 是否允许读取原始 HTML | 默认允许可访问性树和页面语义，不允许直接下载源文件、Source Map、JS Bundle 或读取浏览器缓存 |
| 输出是什么 | 完整代码仓库、依赖说明、Dockerfile/Compose、初始化脚本 |
| 如何判断正确 | 以运行后的产品行为为准，不比较代码相似度 |
| 是否考核后端 | **考核**。有状态的网站必须实现真实后端，不能只做静态页面 |
| 是否允许调用原网站 | 不允许。评测时切断目标网站访问和外部网络 |
| 网站来源 | 自建、可重置、可控的合成网站或白标网站，不直接依赖线上商业网站 |
| v1 输出技术栈 | 先允许自由技术栈；后续增加指定 Python、Rust 等后端语言的约束赛道 |
| 初始数据量 | 先做 5 个手工站点，验证系统，再扩到 20 个基础站点和 100–200 个变体 |
| Human-in-the-loop | 单独设赛道；主赛道默认 Agent 独立完成 |

你备注中的“Fable 5”暂按“首批 5 个手工构建站点”处理。

## **三、Benchmark 任务协议**

### **3.1 Agent 获得的输入**

每个任务包含：

1\. target\_url  
2\. public\_prd  
3\. 测试账号或注册方式  
4\. 可使用的浏览器工具  
5\. 时间、Token、工具调用预算  
6\. 输出协议  
7\. 允许和禁止的访问方式

公开 PRD 只描述产品范围和必须实现的行为，不描述内部技术实现。

例如：

用户可以注册、登录和重置密码。  
用户可以搜索商品、加入购物车并完成模拟结算。  
同一设备或邮箱在五分钟内只能发起一次注册。  
购物车在刷新页面和重新登录后需要保留。

### **3.2 公平性原则**

所有评测要求必须满足至少一个条件：

1. 已在公开 PRD 中声明；  
2. 可以通过正常浏览器探索观察到；  
3. 可以从页面提示、错误信息或用户流程中合理推断。

不应使用 Agent 无法观察、也未在 PRD 中声明的内部规则作为隐藏扣分项。

### **3.3 Agent 输出**

提交物至少包括：

/app  
  frontend/  
  backend/  
  README.md  
  Dockerfile  
  docker-compose.yml  
  seed 或初始化脚本  
  环境变量模板

必须满足：

* 从干净环境可自动安装和编译；  
* 一条命令启动；  
* 不依赖目标网站；  
* 不通过 iframe、反向代理或远程截图伪装；  
* 数据状态可初始化和重置；  
* 评测账号和测试数据可重复生成。

## **四、前端与后端如何设计**

设计原则是：

后端不是单独追求复杂度，而是服务于前端暴露出来的产品能力和业务约束。

每个网站先建立一张“功能—状态—后端—测试”映射表。

| 用户功能 | 前端状态 | 后端能力 | 关键约束 | 评测方式 |
| ----- | ----- | ----- | ----- | ----- |
| 注册 | 表单、错误提示、成功跳转 | 用户创建、唯一性检查 | 五分钟注册限流 | 连续注册、错误信息、时间窗口测试 |
| 登录 | Loading、错误状态、会话状态 | 密码验证、Session | 无效密码、过期会话 | 刷新、重新打开浏览器 |
| 忘记密码 | 请求页面、邮件已发送状态 | Token、邮件服务、过期机制 | Token 一次性、有效期 | 使用旧链接和重复链接 |
| 购物车 | 数量、总价、空购物车 | 持久化购物车 | 登录前后合并规则 | 刷新、重新登录、多窗口 |
| 下单 | 结算过程、成功页 | 订单、库存、事务 | 库存不能为负 | 两个用户购买同一库存 |
| 订单历史 | 列表、详情、状态 | 用户与订单关联 | 用户只能看到自己的订单 | 跨账号访问测试 |

不要求 Agent 复刻参考网站的数据库结构或 API 路径。只要用户可观察行为一致，内部实现可以不同。

### **时间类功能的测试约定**

“五分钟限流”“Token 一小时过期”等规则不应真的让评测等待几分钟或几小时。

Corpus 需要提供统一的可测试时间机制，例如：

* 测试环境中的可控时钟；  
* 通过环境变量缩短测试窗口；  
* 参考网站和候选网站使用相同的测试时间配置。

这项约定必须在任务协议中公开，避免形成不可实现的隐藏测试。

## **五、Benchmark 赛道**

### **5.1 Core Agent Track**

Agent 可以反复：

* 探索目标网站；  
* 编写代码；  
* 启动自己的实现；  
* 对自己的实现进行测试和修改。

期间不允许人工帮助。

这是 v1 的主赛道。

### **5.2 One-shot Track**

主要针对简单网站：

1. Agent 探索目标网站；  
2. 一次性生成代码；  
3. 不能看到编译和评测反馈后再次修改。

用于衡量“一次生成”能力。

### **5.3 Human \+ Agent Track**

人工可以进行有限干预，所有干预都记录：

* 产品理解提示；  
* 调试方向；  
* 代码修改建议；  
* 直接代码编辑；  
* 测试补充。

结果同时报告：

* 最终质量；  
* 人工消息数量；  
* 人工花费时间；  
* 总工具调用和计算成本。

这样可以研究“几小时完成”和“几天完成”的复杂项目，而不把人工贡献混入纯 Agent 分数。

### **5.4 Prescribed Stack Track**

要求候选实现使用指定技术栈，例如：

* React \+ Python；  
* React \+ Rust；  
* 前端 TypeScript \+ 后端 Java；  
* 压力测试赛道中的 C++ 后端。

建议 v1 先使用 Python 和自由技术栈。若 C++ 导致绝大多数任务直接编译失败，它更适合作为后续压力测试，而不是核心排行榜。

### **5.5 Long-context / Memory Stress Track**

专门测量备注中提到的上下文压缩问题：

1. Agent 完成较长时间的网站探索；  
2. 将探索记录压缩成摘要；  
3. 暂时关闭目标网站访问；  
4. Agent 只根据压缩后的记录完成实现；  
5. 检查它是否退回到“模型记忆中的常见网站”，做出一个像但不一致的版本。

Core Track 中仍允许随时回看目标网站；Memory Stress 作为单独实验，不与主赛道混合。

## **六、Corpus 构建方案**

## **6.1 数据量级**

采用“少量基础站点 × 多个受控变体”的方式：

5 个 Pilot 站点  
→ 20 个基础站点  
→ 每个基础站点生成 5–10 个高质量变体  
→ 总计 100–200 个任务

不能只做换 Logo、换颜色的表面变体。每个变体至少改变一个产品逻辑或交互结构。

可变化维度：

* 品牌、文案、图像和数据；  
* 信息架构和导航顺序；  
* 表单字段和验证规则；  
* 业务约束；  
* 页面布局；  
* 交互方式；  
* 加载和异步状态；  
* 用户角色；  
* 后端技术栈；  
* 移动端和桌面端布局。

例如，Amazon-like 站点可以使用完全自建的白标名称和域名，不使用 Amazon 的真实代码、数据和品牌资源；同时更改导航、结算流程和业务规则，以降低模型记忆带来的收益。

## **6.2 网站类型矩阵**

建议覆盖：

| 类型 | 核心能力 |
| ----- | ----- |
| 营销网站 | 响应式布局、Tab、Modal、表单、动画 |
| 内容网站 | 搜索、分页、详情、评论、用户状态 |
| SaaS Dashboard | 登录、CRUD、筛选、图表、权限 |
| 任务管理 | 看板、拖拽、Autosave、多人状态 |
| 预约系统 | 日期、库存、时间冲突、取消 |
| 电商 | 商品、购物车、结算、订单、库存 |
| 社交产品 | Feed、点赞、评论、关注、通知 |
| 客服系统 | 工单、附件、状态流转、角色权限 |
| 文件管理 | 上传、目录、重命名、权限、搜索 |
| Marketplace | 买卖双方、交易状态、评价、消息 |

## **6.3 难度定义**

难度不能只根据代码行数判断，而应根据“可观察状态空间”和“业务流程复杂度”划分。

### **Simple**

* 1–3 个主要页面；  
* 无登录或只有轻量状态；  
* 0–1 个持久化实体；  
* 主要是视觉和简单交互；  
* 强 Agent 可 one-shot 完成。

### **Medium**

* 4–10 个页面或 Route；  
* 有登录和 CRUD；  
* 2–4 个持久化实体；  
* 有异步、错误状态和跨页面状态；  
* Agent 通常需要持续迭代数小时。

### **Hard**

* 多角色或权限系统；  
* 注册、密码重置、通知；  
* 交易、库存或一致性逻辑；  
* 时间、限流或并发规则；  
* 拖拽、无限滚动、跳转、复杂响应式交互；  
* 需要长时间探索和多轮实现。

最终难度应由基线 Agent 的实际完成率、解决时间和错误类型进行校准，而不是只由构建者主观判断。

## **6.4 每个 Corpus Item 必须包含**

1\. 私有参考源代码  
2\. 可部署的参考 Docker Image  
3\. 公开 PRD  
4\. 数据 Seed  
5\. 页面和状态清单  
6\. 用户旅程清单  
7\. 视觉检查点  
8\. 公开 Smoke Tests  
9\. 私有功能测试  
10\. 私有边界条件测试  
11\. 难度和能力标签  
12\. Human \+ Agent 构建轨迹  
13\. 许可证和素材来源

## **七、训练集、验证集和测试集**

不能把同一个基础站点的不同皮肤分别放进训练集和测试集，否则会造成严重模板泄漏。

必须按“网站家族”拆分：

### **Train**

可以公开：

* 参考源代码；  
* PRD；  
* 测试；  
* Human \+ Agent 轨迹；  
* 成功和失败实现。

### **Validation**

公开：

* 目标网站；  
* PRD；  
* 部分测试。

不公开参考源代码和完整隐藏测试。

### **Test**

只开放受控目标网站和任务协议。

以下内容保持私有：

* 源代码；  
* 隐藏测试；  
* 数据随机化规则；  
* 完整评分细节；  
* 未发布的网站变体。

所有同源变体必须处于同一个 Split。

## **八、评测体系**

## **8.1 评测流程**

参考站点启动  
→ 初始化相同数据  
→ Agent 探索并提交代码  
→ 构建候选站点  
→ 运行 Smoke Test  
→ 执行视觉检查点  
→ 执行交互测试  
→ 执行完整用户旅程  
→ 执行状态与边界条件测试  
→ 执行反作弊检查  
→ 生成分数和失败报告

## **8.2 建议评分**

| 维度 | 权重 |
| ----- | ----- |
| 视觉与布局一致性 | 20% |
| 基础交互一致性 | 20% |
| 完整用户旅程和后端行为 | 40% |
| 鲁棒性、异常和边界条件 | 15% |
| 资源效率 | 5% |

其中：

* **编译和启动失败：硬失败；**  
* **发现代理、iframe 或运行时访问目标网站：硬失败；**  
* **核心业务流程完全缺失：该流程计零分。**

不建议把“参考源代码相似度”计入分数，因为同一产品行为可以有多种正确实现。

## **8.3 功能一致性的具体定义**

对于每个测试场景：

1. 将参考网站和候选网站重置到等价初始状态；  
2. 执行同一动作序列；  
3. 比较每一步的页面状态、错误提示和可操作元素；  
4. 比较最终业务状态；  
5. 刷新、重新登录或重启服务后再次检查状态。

重点检查：

* 同一按钮是否在正确状态下出现；  
* 点击后是否进入正确页面；  
* 表单验证是否一致；  
* 错误提示是否表达同一含义；  
* 数据是否持久化；  
* 不同用户是否看到正确的数据；  
* 同一操作能否重复执行；  
* 页面刷新后状态是否正确恢复。

## **8.4 视觉评测**

不能只依赖像素级截图差异。建议组合：

* 截图感知相似度；  
* 元素位置和尺寸；  
* 文本内容；  
* 字体层级；  
* 响应式布局；  
* 人工抽查。

动态数据、时间、随机图片等区域需要 Mask 或固定 Seed。

## **8.5 动态交互专项**

Corpus 中应明确覆盖：

* Sticky Header；  
* Anchor Jump；  
* 无限滚动；  
* Scroll 恢复；  
* Modal 和焦点管理；  
* Hover 状态；  
* 拖拽；  
* 动画结束状态；  
* Loading、Empty、Error 状态；  
* 移动端菜单和响应式断点。

## **九、Trajectory 数据**

Human \+ Agent 完成网站复刻的过程天然可以形成高价值轨迹数据。

每次任务记录：

task\_id  
目标网站和版本  
公开 PRD  
Agent 观察结果  
Browser actions  
截图或可访问性树  
Agent 消息  
Shell 和编译输出  
代码 Diff  
候选网站测试结果  
失败测试  
Human intervention  
干预类别  
最终分数  
资源消耗

### **Human intervention 分类**

* 产品理解；  
* 探索策略；  
* 前端布局；  
* 后端建模；  
* Debug 定位；  
* 代码修改；  
* 测试设计；  
* 发现遗漏功能；  
* 纠正“凭记忆实现”。

最终可得到三类数据：

1. 成功构建轨迹；  
2. 失败后恢复轨迹；  
3. 未能恢复的失败轨迹。

这些数据可以用于：

* SFT；  
* Agent 规划；  
* Tool-use 学习；  
* Critic 或 Reward Model；  
* 自动修复；  
* Human-in-the-loop 研究。

## **十、Failure Corpus**

“先尽量复刻正确，再人为加入一个错误”适合作为单独的 Failure 子集，而不应污染主 Benchmark 的 Golden Truth。

建议命名为：

**Web2Code2Web-Failures**

制作方式：

1. 先得到一个高质量复刻版本；  
2. 每次只注入一个可控错误；  
3. 标注错误位置、影响流程和正确修复方式；  
4. 让模型执行发现、定位和修复。

错误类型包括：

| Failure Mode | 示例 |
| ----- | ----- |
| Memory substitution | 按印象做成常见电商，而非目标站点实际流程 |
| Long-context omission | 忘记早期探索到的特殊验证规则 |
| Frontend-only shortcut | UI 看似正确，但刷新后所有数据丢失 |
| Cache leakage | 直接复用浏览器缓存中的目标资源 |
| Validation mismatch | 邮箱、密码、数量等校验条件错误 |
| Session mismatch | 登录状态、退出或多账号隔离错误 |
| Time-rule mismatch | 五分钟限流或 Token 过期逻辑错误 |
| State persistence | 购物车、订单或表单无法持久化 |
| Dynamic interaction | 滚动、拖拽、跳转或 Modal 行为错误 |
| Familiar-layout drift | 做成“像 Amazon”，但不像实际白标目标站 |
| Concurrency failure | 库存、重复提交或多用户状态不一致 |
| Visual near miss | 主体正确但间距、层级、响应式持续偏差 |

## **十一、反作弊设计**

### **11.1 防止直接拿到前端**

* 参考源代码不挂载到 Agent 环境；  
* 关闭 Source Map；  
* 浏览器容器和代码容器隔离；  
* Agent 无法读取浏览器缓存目录；  
* 不允许从 Shell 直接请求目标站点；  
* 目标访问只能经过受控 browser-use Gateway；  
* browser-use 不暴露 Network Response Body 和 Bundle 文件；  
* 下载功能默认关闭。

需要注意：浏览器要渲染页面，本身必然会接收 HTML、CSS 和 JavaScript。真正可执行的边界应是：

允许浏览器完成正常渲染和语义观察，但不允许 Agent 将目标前端资源直接导出到代码工作区。

### **11.2 防止代理和 iframe**

评测候选网站时：

* 完全关闭目标站点访问；  
* 默认关闭外部网络；  
* 扫描目标域名和 iframe；  
* 记录运行时网络请求；  
* 检查是否包含参考 Bundle、资源 Hash 或大段复制代码；  
* 使用一次性目标域名，避免硬编码。

### **11.3 防止 Save Cache**

* 每次运行使用新的浏览器 Profile；  
* 浏览器缓存不挂载；  
* 目标 URL 加入 Run-specific Token；  
* 每次任务更换临时域名；  
* 浏览器和 Agent 文件系统完全分离。

### **11.4 防止截图式伪装**

* 使用动态数据；  
* 修改初始 Seed；  
* 检查真实点击目标；  
* 检查表单输入和状态变化；  
* 在不同视口运行；  
* 用多账号、多会话测试；  
* 对关键数据执行随机化。

### **11.5 防止硬编码测试数据**

* 随机用户名、商品和订单；  
* 随机测试执行顺序；  
* 多组 Seed；  
* 测试账号和业务数据在运行时生成；  
* 同一流程使用不同输入重复执行。

---

## **十二、反 Memory 设计**

为了避免模型仅凭“它知道哪些网站”完成任务：

1. 使用白标品牌和自定义域名；  
2. 使用合成数据；  
3. 对熟悉产品模式进行结构扰动；  
4. 加入与常见网站不同、但产品上自洽的规则；  
5. 设置“看起来熟悉、实际流程不同”的站点；  
6. 测试集使用未公开变体；  
7. 不在任务名称中出现 Amazon、Twitter、Notion 等强提示词；  
8. 训练集和测试集按产品家族隔离。

建议建立三个对照基线：

| Baseline | 输入 |
| ----- | ----- |
| Memory Baseline | 只有产品简介，没有浏览器访问 |
| Browser Baseline | 产品简介 \+ browser-use |
| Oracle Baseline | 产品简介 \+ 参考源代码，仅作上限，不进入排行榜 |

`Browser Baseline - Memory Baseline` 的差距可以衡量探索本身带来的收益。

## **十三、基础设施**

推荐的系统结构：

Reference Site Registry  
        ↓  
Reference Site Runner  
        ↓  
Controlled Browser Gateway  
        ↓  
Agent Runner ─────→ Trajectory Logger  
        ↓  
Candidate Repository  
        ↓  
Candidate Builder  
        ↓  
Candidate Site Runner  
        ↓  
Visual \+ Functional Evaluator  
        ↓  
Score / Failure Report / Artifacts

### **核心组件**

1. **Site Registry**  
   * 网站版本；  
   * PRD；  
   * Seed；  
   * 测试配置；  
   * 难度和能力标签。  
2. **Reference Runner**  
   * 按任务启动目标网站；  
   * 重置数据库；  
   * 提供一次性 URL。  
3. **Browser Gateway**  
   * 只暴露允许的浏览器动作；  
   * 屏蔽源文件导出和 Network Inspection；  
   * 记录所有浏览器轨迹。  
4. **Agent Runner**  
   * 隔离代码工作区；  
   * 控制时间、Token、命令和网络；  
   * 保存完整日志。  
5. **Candidate Builder**  
   * 从干净环境安装；  
   * 编译；  
   * 启动；  
   * 执行健康检查。  
6. **Evaluator**  
   * 自动化用户旅程；  
   * 截图；  
   * 状态测试；  
   * 多用户和多会话测试；  
   * 网络和反作弊检查。  
7. **Artifact Store**  
   * 代码；  
   * 截图；  
   * 视频；  
   * Trace；  
   * 测试报告；  
   * 失败分类。

### **“半天 Infra”的合理范围**

半天可以完成：

* 一个参考站点；  
* 一个 Agent 容器；  
* 一个候选站点容器；  
* 一条 Playwright Smoke Test；  
* 基础日志。

但以下部分需要继续工程化：

* 浏览器和代码环境隔离；  
* 反作弊；  
* 可重复 Seed；  
* 隐藏测试；  
* 多账号状态；  
* 截图稳定性；  
* 资源和网络控制；  
* 100–200 个站点的批量调度。

因此应把“半天”定义为骨架 MVP，而不是完整 Benchmark Infra。

## **十四、首批 5 个 Pilot 站点**

| 站点 | 难度 | 核心目标 |
| ----- | ----- | ----- |
| 1\. Interactive Landing | Simple | 响应式、Tab、Modal、表单、滚动动画 |
| 2\. Product Catalog | Simple/Medium | 搜索、筛选、排序、详情、URL 状态 |
| 3\. Task Manager | Medium | 注册、登录、CRUD、持久化、状态筛选 |
| 4\. Booking App | Medium | 日期选择、库存、冲突、取消、错误状态 |
| 5\. White-label Commerce | Hard | 注册、找回密码、搜索、购物车、结算、订单、限流 |

第五个站点可以承接你备注中的 Amazon-like 案例，但应使用自有名称、素材、数据和流程。

## **十五、项目里程碑**

### **Phase 0：协议冻结与骨架**

**目标：跑通一个端到端任务。**

交付：

* Benchmark Task Contract；  
* Agent 权限规则；  
* Site Manifest；  
* 参考站点容器；  
* Agent → Code → Candidate 部署；  
* 1 条视觉测试；  
* 1 条完整功能测试；  
* 基础 Trace。

完成标准：

* Agent 无法看到参考源代码；  
* 候选代码可以从干净环境构建；  
* 目标和候选可使用同一测试场景；  
* 评测结果可重复。

### **Phase 1：5-site Pilot**

**目标：验证题目设计和难度分层。**

交付：

* 5 个站点；  
* 每站 5–15 条用户旅程；  
* 自动化测试；  
* 20 条以上 Agent 或 Human \+ Agent 轨迹；  
* 初版反作弊；  
* Memory、Browser、Oracle 三个基线。

Go/No-Go 标准：

* 不是所有模型都满分，也不是全部归零；  
* 分数能够区分简单、中等和困难站点；  
* 同一提交重复评测结果稳定；  
* 隐藏测试没有不可观察要求；  
* 没有明显源代码泄漏路径。

### **Phase 2：20 个基础站点**

**目标：形成覆盖主要产品形态的基础 Corpus。**

交付：

* 20 个基础站点；  
* 产品类型矩阵；  
* 统一 Seed 和时间机制；  
* Trajectory Schema；  
* Failure Taxonomy；  
* 自动质量检查；  
* 家族级 Train/Validation/Test Split。

### **Phase 3：扩到 100–200**

**目标：验证 Scale-up。**

交付：

* 受控变体生成器；  
* 每个基础站点 5–10 个变体；  
* 自动部署和测试；  
* 人工二次验收；  
* 去重和泄漏检查；  
* 批量基线运行。

### **Phase 4：Benchmark Release**

交付：

* Benchmark Card；  
* 数据说明；  
* 评分协议；  
* Baseline Report；  
* 公开训练集；  
* 私有测试服务；  
* Leaderboard 或内部评测面板；  
* Failure Corpus。

## **十六、角色与责任**

| 角色 | 责任 |
| ----- | ----- |
| Research / Product Lead | 冻结任务定义、PRD、难度和研究问题 |
| Infra Engineer | Runner、Browser Gateway、隔离、调度、日志 |
| Evaluation Engineer | 用户旅程、视觉评测、隐藏测试、反作弊 |
| Corpus Full-stack Builder | 实现参考网站和变体 |
| Human Annotator | 探索网站、验证 PRD、采集 HITL 轨迹 |
| QA Reviewer | 独立验收、发现不可观察测试和数据泄漏 |

每个 Corpus Item 至少经过：

1. 构建者自测；  
2. 独立人员仅根据公开 PRD 和网站进行验收。

## **十七、主要风险**

| 风险 | 缓解 |
| ----- | ----- |
| Agent 直接复制前端资源 | 浏览器与代码隔离、关闭缓存访问、评测断网、资源 Hash 检查 |
| 视觉很好但没有后端 | 多会话、刷新、重启、多用户和随机数据测试 |
| 隐藏要求无法发现 | 要求每个测试都能由 PRD 或正常探索支持 |
| 变体只是换皮 | 每个变体必须修改至少一个流程或业务规则 |
| 测试集模板泄漏 | 按网站家族拆分，而非按页面实例拆分 |
| 难题全部失败 | 用 Pilot 基线校准，避免核心集出现大面积零分 |
| 简单题全部满分 | 增加动态状态、边界条件和随机化 |
| 截图评测不稳定 | 固定字体、Seed、视口，Mask 动态区域，组合结构指标 |
| Human 帮助不可比较 | 限制干预预算并完整记录干预类型和时间 |
| 模型凭记忆完成 | 白标网站、结构扰动、无品牌提示、Memory Baseline 对照 |
| Long-context 后遗漏细节 | 单独建立 Memory Stress Track 和相关 Failure Corpus |

## **十八、可以直接进入开发的 P0 Backlog**

1. 写出 `task_contract.md`，冻结输入、输出和禁止项。  
2. 定义 `site_manifest.yaml`。  
3. 搭建 Reference、Agent、Candidate、Evaluator 四个容器。  
4. 实现 browser-use 受控入口，隔离浏览器缓存。  
5. 完成第一个 Task Manager 参考站点。  
6. 为其编写注册、登录、CRUD、刷新持久化四条流程。  
7. 跑通 Agent 探索、代码生成、重新部署和自动评测。  
8. 保存浏览器动作、消息、代码 Diff 和测试结果。  
9. 加入一个只做前端假状态的作弊版本，确认评测能识别。  
10. 加入一个 iframe/代理版本，确认反作弊能阻断。  
11. 完成 5-site Pilot。  
12. 根据 Pilot 结果再冻结 100–200 Corpus 的生成模板。

**最优启动顺序不是先批量做网站，而是先用一个中等难度、带登录和持久化的站点，把 Web → Code → Web → Evaluation 的闭环彻底跑通。**
